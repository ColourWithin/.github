outputs:

**Runs block — single step, id must be "exchange":**

```yaml
runs:
  using: composite
  steps:
    - id: exchange
      name: OCI token exchange
      shell: bash
      env:
        INPUT_CLIENT_IDENTIFIER: ${{ inputs.client-identifier }}
        INPUT_CLIENT_SECRET: ${{ inputs.client-secret }}
        INPUT_DOMAIN_BASE_URL: ${{ inputs.domain-base-url }}
        INPUT_AUDIENCE: ${{ inputs.audience }}
        INPUT_REGION: ${{ inputs.region }}
        INPUT_OUTPUT_CONFIG_PATH: ${{ inputs.output-config-path }}
        INPUT_OUTPUT_KEY_PATH: ${{ inputs.output-key-path }}
      run: |
        # Preflight: fail fast before incurring pip install cost (D-12)
        # This check intentionally runs before pip install — a misconfigured workflow
        # fails in <1s rather than waiting 5-8s for the install to complete
        if [[ -z "${ACTIONS_ID_TOKEN_REQUEST_URL:-}" || -z "${ACTIONS_ID_TOKEN_REQUEST_TOKEN:-}" ]]; then

          echo "::error::permissions:"
          echo "::error::  id-token: write"
          echo "::error::  contents: read"
          exit 1
        fi

        pip install --user 'oci>=2.173.1,<3' requests cryptography

        python "${{ github.action_path }}/exchange.py"
```

**Critical constraints to check:**

1. The `shell: bash` line specifies bash. GitHub Actions composite actions inherit the default shell behaviour — without `shell:` it would default to `sh` on some runners. Explicitly set `bash`.

2. The `run:` block does NOT include `set -x`. Shell runs with the implicit behaviour of `bash -e -o pipefail` from the `shell: bash` setting in composite actions. Do not add explicit `-e -o pipefail` headers — they are applied by the runner. What matters is that `set -x` is ABSENT (TOKEX-14).

3. `${{ github.action_path }}` in the `run:` body is SAFE and intentional — it is a runner-provided context expression, not a user-supplied input. The RESEARCH.md explicitly verifies this is the correct pattern.

4. ALL inputs flow via the `env:` block as `INPUT_*` env vars. There must be NO `${{ inputs.foo }}` references inside the `run:` body itself (only in the `env:` block is acceptable per D-08).

5. The step `id:` is `exchange` — this must exactly match the output references in the `outputs:` block above (`${{ steps.exchange.outputs.config-path }}`).

6. `${{ runner.home }}` in the `default:` fields is correct — this is a YAML expression context, not a shell variable context. Do NOT use `$HOME` or `~` in `default:` fields.

**File 1: `actions/oci-token-exchange/README.md`**

The README must contain exactly these 10 sections (in this order). Content requirements for each are specified below:

**Section 1: What this action does (1 paragraph)**
State: exchanges a GitHub Actions OIDC ID token for an OCI User Principal Session Token (UPST) via Oracle's `oci.auth.signers.TokenExchangeSigner` (Option A — composite + Python). Mention that it replaces `gtrevorrow/oci-token-exchange-action` which ships unpatched `axios < 1.12.0` (CVE-2025-58754) and produces a UPST with `principal.type = 'user'` rather than `'workload'`.

**Section 2: Prerequisites**
- OCI Identity Propagation Trust (IPT) configured to trust `https://token.actions.githubusercontent.com`
- OCI Confidential Application with OAuth client credentials
- Service User mapped via IPT policy
- `OCI_OIDC_CLIENT_IDENTIFIER` and `OCI_OIDC_CLIENT_SECRET` populated in the consuming repo's GitHub Secrets
- `OCI_DOMAIN_BASE_URL` populated in the consuming repo's GitHub Variables
- Note: IPT and Service User are built via Tofu in `colour-within-ops/deploy/tofu/modules/identity/`

**Section 3: Required consumer permissions**
Include this exact YAML block verbatim:
```yaml
permissions:
  id-token: write
  contents: read
```
State that without `id-token: write`, no GitHub OIDC token can be minted and the action will fail immediately with an actionable error message naming this exact fix.

**Section 4: Inputs / outputs table**
Two tables sourced from `action.yml`:

Inputs table with columns: Name | Required | Default | Description
- `client-identifier` | yes | — | OAuth client_id (from `secrets.OCI_OIDC_CLIENT_IDENTIFIER`)
- `client-secret` | yes | — | OAuth client_secret (from `secrets.OCI_OIDC_CLIENT_SECRET`)
- `domain-base-url` | yes | — | Identity Domain base URL
- `audience` | no | `https://github.com/ColourWithin` | OIDC audience
- `region` | no | `ap-sydney-1` | OCI region
- `output-config-path` | no | `~/.oci/config` | Path for OCI CLI config
- `output-key-path` | no | `~/.oci/upst.pem` | Path for ephemeral RSA private key

Outputs table with columns: Name | Description
- `config-path` | Absolute path to written OCI CLI config
- `expires-at` | ISO 8601 UPST expiry timestamp

**Section 5: Usage example**
A complete minimal consumer workflow snippet demonstrating SHA-pinned consumption. Use a placeholder SHA (`<sha>`) since the real v1.0.0 SHA is not yet known. Include the doubled `.github` path in a comment. Example:

```yaml
jobs:
  deploy:
    runs-on: ubuntu-24.04
    permissions:
      id-token: write
      contents: read
    steps:
      - name: Exchange OIDC token for OCI UPST
        # Note: the doubled .github/ path is required — GitHub's org-default-repo rule
        uses: ColourWithin/.github/actions/oci-token-exchange@<sha>
        with:
          client-identifier: ${{ secrets.OCI_OIDC_CLIENT_IDENTIFIER }}
          client-secret: ${{ secrets.OCI_OIDC_CLIENT_SECRET }}
          domain-base-url: ${{ vars.OCI_DOMAIN_BASE_URL }}

      - name: Run OCI CLI command
        run: oci iam region list
        # OCI_CLI_AUTH=security_token is automatically available from the previous step
```

**Section 6: Audience mapping**
State that the `audience` input default is `https://github.com/ColourWithin` and this value MUST be configured in the IPT's `allowedTokenAudiences` field. Warn that Oracle's own SDK example (`workload_identity_federation_signer_example.py`) uses `"github-actions"` as a placeholder — do not copy that value; use the real audience URL configured on the IPT.

**Section 7: `requested_token_type` literal correction**
State that the OCI PRD §3 shows `urn:oci:token-type:upst` but the canonical OCI literal is `urn:oci:token-type:oci-upst` (with the `oci-` infix). The SDK uses the correct literal internally. This is documented here to prevent confusion if consumers compare the PRD to the SDK source.

**Section 8: UPST lifetime ceiling**
State: each exchange produces a UPST with a ~60-minute lifetime. The OCI CLI does not refresh session tokens during execution (see [oracle/oci-cli#998](https://github.com/oracle/oci-cli/issues/998)). For jobs that may exceed ~55 minutes, re-invoke this action between long phases. Do not assume the UPST will be automatically refreshed.

**Section 9: Filesystem footprint**
The action writes three files to the runner's filesystem (paths shown for default inputs):
- `~/.oci/config` — OCI CLI `[DEFAULT]` profile stanza
- `~/.oci/upst.token` — UPST JWT text (same directory as config; basename always `upst.token`)
- `~/.oci/upst.pem` — Ephemeral RSA-2048 private key (PEM, chmod 600)

Note: `OCI_CLI_AUTH=security_token` is exported to all subsequent steps in the same job automatically.

**Section 10: Self-hosted runner warning**
Composite actions have no `post:` lifecycle hook (GitHub Actions limitation). On self-hosted runners, the three files above will persist on disk after the job completes until the runner VM is recycled or manually cleaned up. This is acceptable for GitHub-hosted ephemeral runners (VMs are discarded after each job) but should be considered for self-hosted runner configurations.
